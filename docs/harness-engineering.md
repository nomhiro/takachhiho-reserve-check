# ハーネスエンジニアリング設計書 — takachiho-watch

「ハーネス」とは、Claude Code がこのプロジェクトを安全かつ確実に実装・運用するための
**足場（scaffolding）** を指す。具体的には、開発ループ・テスト戦略・観測性・段階的ロールアウト・
受け入れ基準の設計を含む。

## 1. 設計思想

TPS の観点で整理すると以下の3点に集約される:

| TPS概念 | このプロジェクトでの実現 |
|---|---|
| **自働化（じどうか）** | 異常時はジョブが自分で気付き通知する。サイレントな失敗を許さない |
| **見える化** | state.json の git 履歴 + 構造化ログで「いつ何が起きたか」が常に追える |
| **正味率向上** | 不要な通知（毎時通知）を排し、価値のある状態変化のみを通知する |

## 3. テスト戦略

### 3.1 ユニットテスト

#### parser
| ケース | 入力 | 期待 |
|---|---|---|
| 空きありHTML | fixtures/available.html | `"available"` |
| 満室HTML | fixtures/full.html | `"full"` |
| 空レスポンス | "" | 例外 raise |
| 想定外構造 | fixtures/error.html | 例外 raise |

#### state machine
| from | to | 期待アクション |
|---|---|---|
| unknown | full | サイレント、state更新 |
| unknown | available | **urgent通知**、state更新 |
| full | full | サイレント、last_check_atのみ更新 |
| full | available | **urgent通知**、state更新 |
| available | available | サイレント、last_check_atのみ更新 |
| available | full | default通知、state更新 |
| any | error（連続3回） | high通知、consecutive_errors=0 にリセット後 error 状態へ |
| error | available | **urgent通知**、state更新 |
| error | full | low「復旧」通知、state更新 |

#### notifier
- urgent / default / low / high のペイロードが仕様書通りであること
- `--dry-run` 時に実際のHTTP送信をしないこと

### 3.2 結合テスト

`scripts/check.py` を fixture とモックHTTPで動かし、state.json の変遷を検証。

### 3.3 E2Eテスト（手動）

テストトピックを購読したスマホで以下を確認:

1. state.json を `{"last_status": "full"}` に書き換え → 実行 → 通知なし
2. state.json を `{"last_status": "full"}` に書き換え + parser をモックで `available` 返却 → 実行 → urgent通知が届く
3. urgent通知の音が他のアプリと違うこと（ntfyアプリで設定可能）
4. 通知タップで予約ページが開くこと

## 4. 観測性

### 4.1 ログ

- 構造化ログ（JSON Lines）を stdout に出力
- GitHub Actions UI で各実行の判定結果が追える

### 4.2 状態の可視化

`state/state.json` のコミット履歴がそのまま「監視ログ」になる:

```bash
git log --oneline state/state.json
# chore(state): update at 2026-04-28T07:05Z
# chore(state): update at 2026-04-28T08:05Z
# ...
```

差分を見れば状態変化が一目で分かる:

```bash
git log -p state/state.json
```

### 4.3 ヘルスチェック

- 毎週日曜 9:00 JST にntfyへ「稼働中」通知
- ユーザーがこの通知を受け取らない週は異常 → Actions UI で確認

### 4.4 失敗検知

- ジョブ自体が失敗した場合、GitHub の通知設定で個人にメールが飛ぶ（GitHub > Settings > Notifications で確認）

## 5. 段階的ロールアウト

| Phase | 内容 | 終了条件 |
|---|---|---|
| **Phase 0: 準備** | リポジトリ作成、Secrets 設定、ntfyアプリインストール | テストトピックを購読してテスト送信が届く |
| **Phase 1: dry-run** | `workflow_dispatch` で手動実行、通知抑止モード | 3回連続で正しい状態判定がログに出る |
| **Phase 2: テストトピック** | `NTFY_TOPIC` をテスト用に設定、cron 有効化 | 24時間動かして異常なし、ヘルスチェック通知到達 |
| **Phase 3: 本番** | `NTFY_TOPIC` を本番用に切り替え | 5/8 まで継続稼働 |

## 6. 受け入れ基準（Definition of Done）

実装完了の判定基準:

- [ ] `docs/` 配下の3ドキュメントを読んだ上で実装している
- [ ] サイトの実装形式（SSR/CSR）が事前調査済み
- [ ] `pytest` が全パス
  - [ ] parser: 4ケース以上
  - [ ] state machine: 遷移マトリクスの全セル
  - [ ] notifier: 4種の通知ペイロード
- [ ] `python scripts/check.py --dry-run` が実サイトに対して成功する
- [ ] テストトピックで4種の通知（urgent/default/low/high）すべてスマホで受信確認
- [ ] urgent通知のみ専用音/バイブで他と区別できる
- [ ] state.json が PR 内で適切に更新される
- [ ] workflow yaml が cron + 手動実行 + 5/8以降の自動停止に対応
- [ ] README に運用手順とトラブルシューティングが記載されている
- [ ] CLAUDE.md とコードに矛盾がない（実装後に CLAUDE.md を最新化）

## 7. 想定リスクと対処

| リスク | 影響 | 対処 |
|---|---|---|
| サイト構造変更 | パース失敗、通知漏れ | 連続3回失敗で監視エラー通知。ヘルスチェック未着で気付ける |
| ntfy.sh 自体がダウン | 通知漏れ | 不可避。重要な日（5/8直前）はヘルスチェックを毎日に増やす運用検討 |
| GitHub Actions の cron 大幅遅延 | 通知遅延 | 1時間ごとなので最大75分の遅延は許容。重要日はcronを30分間隔に短縮検討 |
| state.json コミット競合 | ジョブ失敗 | `concurrency` で同時実行を防止済み |
| トピック名漏洩 | 第三者から偽通知 | 推測困難な文字列を使用。漏洩疑い時は即座にトピック変更 |
| 5/8 直前にバグ発覚 | 通知機会喪失 | Phase 2 で十分な期間（最低3日）動作確認する |

## 8. 撤収手順

5/8 経過後:

```bash
# ワークフロー無効化
gh workflow disable watch.yml

# またはリポジトリごとアーカイブ
gh repo archive
```

ntfy トピックの購読は手動でアプリから解除。
